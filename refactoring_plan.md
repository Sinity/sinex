### **Part 27, Addendum: Final Architectural Refinements**

This addendum proposes key modifications and additions to the previously outlined refactoring roadmap.

#### **Refinement 1: The "Event Topic" Strategy for Redis Streams**

**Problem:** The proposal to have topics like `sinex:events:fs` and `sinex:events:shell.kitty` is too granular. It would lead to dozens or hundreds of streams, and automata would need to subscribe to many of them, which is inefficient.

**Refined Solution: A Single, Unified "Hot Log" Stream.**

* **Action:** Modify `sinex-ingestd` to publish **all** events to a single, high-throughput Redis Stream named **`sinex:streams:hotlog`**.
* **How it Works:** Each message in the stream will contain the full `RawEvent` or `SynthesisEvent` as a serialized payload (e.g., MessagePack or JSON). The event's `source` and `event_type` fields are inside this payload.
* **Automaton Filtering:** Automata will **not** subscribe to specific Redis topics. Instead, they will all subscribe to the single `hotlog` stream as part of the same consumer group. The filtering logic moves from the message bus (Redis) into the automaton satellite itself.
  * An automaton's `main` loop will read a batch of messages from the `hotlog`.
  * It will then perform a fast, in-memory filter on the batch to select only the events it cares about based on `source` and `event_type`.
* **Benefits:**
  * **Massive Simplification:** Radically simplifies the `sinex-ingestd` logic and the message bus topology. There is only one stream to manage.
  * **Future-Proofing:** When a new event source is added, no existing automata need to change their subscriptions. They will simply ignore the new event types by default.
  * **Performance:** A single stream with multiple consumer groups is a highly optimized pattern in Redis. It's more efficient than managing many small streams.

#### **Critical Addition: A Formal "Replay" and "Backfill" API**

**Problem:** We've discussed replaying automata and backfilling historical data, but the mechanism for triggering and managing this is ad-hoc. This needs to be a first-class feature of the architecture.

**Solution: The `sinex-orchestrator.service` and the `core.jobs` Table.**

* **Action:** Introduce a new, high-level satellite, the **`sinex-orchestrator`**. This is the brain for all non-real-time, batch processing tasks.
* **Action:** Create a new database table, **`core.jobs`**.

    ```sql
    CREATE TABLE core.jobs (
        job_id ULID PRIMARY KEY,
        job_type TEXT NOT NULL, -- e.g., 'replay_automaton', 'scan_source', 'backfill_embeddings'
        target_name TEXT NOT NULL, -- The name of the automaton or ingestor to run
        parameters JSONB, -- e.g., {"since": "...", "until": "..."}
        status TEXT NOT NULL DEFAULT 'pending', -- pending, running, completed, failed
        created_at TIMESTAMPTZ DEFAULT now(),
        started_at TIMESTAMPTZ,
        completed_at TIMESTAMPTZ,
        result_summary JSONB
    );
    ```

* **The Workflow:**
    1. **Job Creation:** When a user wants to replay an automaton or scan a historical source, they do **not** run the satellite binary directly. Instead, they use the CLI to create a job record:
        `exo jobs create --type replay_automaton --target canonical-command-synthesizer --params '{"since": "3d_ago"}'`
        This simply inserts a new row into the `core.jobs` table.
    2. **Orchestrator Action:** The `sinex-orchestrator` polls the `core.jobs` table for `pending` jobs.
    3. When it finds one, it is responsible for **dynamically starting a one-off systemd service** to execute the task. It can use systemd's transient unit feature (`systemd-run`) for this.
        `systemd-run --unit=sinex-replay-job-ULID --service-property="User=sinex" /path/to/satellite/binary scan --from-job-id JOB_ULID`
    4. **Job Execution:** The satellite binary, when started with `--from-job-id`, reads its parameters from the `core.jobs` table, performs its work (scanning `raw.events` or a historical source), and writes its output to `sinex-ingestd`. It periodically updates its status in the `core.jobs` table.
    5. **Completion:** When the task is done, it updates its job status to `completed` or `failed` and exits. The transient systemd unit is automatically cleaned up.
* **Benefits:**
  * **Robustness:** All historical processing is managed, queued, and tracked transactionally. You never lose a replay or scan job.
  * **Observability:** You have a complete history of all replay and backfill operations in the `core.jobs` table.
  * **Resource Management:** The orchestrator can control how many batch jobs run concurrently, preventing system overload.
  * **Decoupling:** The user's request is decoupled from the execution. The user can fire off a huge backfill job and the orchestrator will ensure it eventually runs.

### **Part 28: The Sinex System - A Gears-Level Architectural Blueprint**

#### **I. The Core Ontology (The "What")**

The Sinex system understands the world through four fundamental concepts:

1. **Event:** An immutable record of a discrete, time-stamped fact. This is the atomic unit of information.
    * **Raw Events (`raw.events`):** Direct, untransformed observations of the external world from Ingestors. Their `source_event_ids` is `NULL`. They are the **Ground Truth**.
    * **Synthesis Events (`synthesis.events`):** Derived, higher-level facts created by Processors. Their `source_raw_event_ids` links them back to the evidence they were derived from. They are the **System's Conclusions**.

2. **Artifact (`core.artifacts`):** A durable, versioned, user-facing knowledge object. Artifacts are the primary "nouns" the user creates and interacts with (e.g., a note, a webpage archive, a project plan). They are created and modified as a *result* of event processing.

3. **Entity (`core.entities`):** A canonical node in the knowledge graph. Represents a unique, real-world concept (e.g., the person "John Doe", the project "Sinex", the topic "NixOS").

4. **Provenance:** A first-class, structural link (`source_raw_event_ids`) that makes the entire system auditable and traceable. It is the causal thread connecting all data.

#### **II. The System's Moving Parts (The "Gears")**

    * **Processors (Automata & Agents):** The intelligence layer. They consume event streams and produce `Synthesis Events` and updates to the `core` tables.
        * **Automata:** Deterministic processors whose logic is defined declaratively (e.g., via `.flow.yaml` files).
        * **Agents:** Imperative processors for complex, non-deterministic, or external-facing tasks (e.g., LLM interaction, paste detection heuristics).

2. **The Hubs (The Nucleus):** Two lean, special-purpose services forming the system's core.

### **Part 28, Addendum: Restoring High-Resolution Details to the Blueprint**

#### **1. The Mechanics of Automaton Replay & The `sinex-orchestrator`**

The blueprint mentions that re-computation is possible, but our discussion settled on a more robust and observable mechanism than just a developer running a satellite with a `--replay` flag.

* **Restored Detail:** The `sinex-orchestrator.service` and the `core.jobs` table are the formal mechanism for this.
  * **The Workflow:** A user or system process never directly invokes a satellite in `replay` or `scan` mode. Instead, it creates a job request by `INSERT`ing a row into `core.jobs`. This request specifies the `job_type` (`replay_automaton`, `scan_source`), the `target_name` (e.g., `canonical-command-synthesizer`), and `parameters` (like a time range).
  * **The Orchestrator:** The `sinex-orchestrator` is a dedicated satellite that polls this table. It acts as a master scheduler, picking up pending jobs and executing them. For true process isolation, it can use `systemd-run` to spawn transient service units for each job.
  * **Benefits:** This provides queuing, observability (you can see the history of all batch jobs), and resource management (the orchestrator can limit how many intensive `replay` jobs run concurrently).

#### **2. The Interactive `explore` Command - The User's Assurance Tool**

The blueprint mentions the `explore` subcommand, but the details are what make it powerful.

* **Restored Detail:** The `explore` command is not just for viewing data; it's an **interactive data assurance and diagnostics TUI**. When a user runs, for example, `sinex-atuin-ingestor explore`, it performs three distinct functions:
    1. **Source Preview:** It directly reads the "external world" source (e.g., the Atuin SQLite DB) to show the user the raw, current state *before* it enters Sinex. This answers the question: "What *could* be ingested?"
    2. **Archive History:** It queries the Sinex PostgreSQL database (`raw.events`) to show what *has been* ingested from that source. This answers: "What *did* I ingest?"
    3. **Coverage Analysis (The Diff):** This is the most critical function. It programmatically compares the Source Preview and the Archive History to identify gaps. It explicitly tells the user, "There are 50 commands in your Atuin DB from yesterday that are not in Sinex." It then offers a one-click action to trigger a targeted `scan` job (via the `core.jobs` table) to fill precisely those gaps.

#### **3. The Hybrid Schema Management Workflow**

The blueprint mentions "GitOps Schema Management," but the specific, pragmatic hybrid workflow we designed is key.

* **Restored Detail:** The process is a fusion of "Code-as-Truth" and "Git-as-Contract":
    1. **Developer Experience (Code-as-Truth):** The developer defines event payloads as simple Rust structs using `#[derive(Serialize, JsonSchema)]` in the `sinex-events` crate. This is the single source of truth for the *implementation*.
    2. **CI Enforcement (Git-as-Contract):** The CI/CD pipeline contains a critical job that:
        a. Automatically **generates the `.json` schema files** from the Rust structs.
        b. **Commits these generated files** back to the `/schemas` directory in the repository.
        c. This ensures the Git repository always contains a canonical, language-agnostic representation of the schema that is guaranteed to be in sync with the Rust code. This is the contract for non-Rust satellites and tools.
    3. **Deployment:** A deployment script or the `sinex-flow-engine` itself reads these validated `.json` files from the Git repo and syncs them into the `sinex_schemas.event_payload_schemas` table for runtime use.

#### **4. The Nature of "Service Automata" and the API Request/Response Loop**

The blueprint mentions that `sinex-gateway` translates API calls into events, but the mechanics are subtle and important.

* **Restored Detail:** "Services" are indeed a subclass of Automata.
    1. A user's API call (`exo find ...`) hits `sinex-gateway`.
    2. The gateway generates a unique `request_id` and publishes a command event (e.g., `api.command.search`) to the `hotlog` stream. The payload includes the query parameters and the `request_id`.
    3. A specific "Service Automaton" (e.g., `sinex-search-service`) is the sole subscriber to this command topic.
    4. It performs the work (querying the database).
    5. It then publishes a response event (e.g., `api.response.search_result`). The key is that the **response topic is specific to the request**: `api:response:REQUEST_ID`.
    6. The `sinex-gateway`, which made the initial request, is the *only* component listening on that unique, ephemeral response topic. It receives the data, matches the `request_id`, and sends the result back to the original client.
    7. This pattern turns a synchronous API call into a fully-auditable, asynchronous, message-passing workflow, making the system incredibly robust and decoupled.

#### **5. The Journald Heartbeat Pattern**

This was an elegant simplification we discussed that is worth codifying in the blueprint.

* **Restored Detail:** Instead of having every satellite implement custom heartbeat logic, they simply log structured JSON messages to `stdout`.
  * `{"level": "INFO", "message": "heartbeat", "status": "healthy", ...}`
  * `systemd` captures this `stdout` and writes it to the system journal.
  * The `sinex-journald-ingestor` captures this as a `raw.events` entry.
  * A single, simple Automaton (`sinex-health-aggregator`) subscribes to these specific journal events and updates the overall system health view. This radically simplifies the implementation of every other satellite.
