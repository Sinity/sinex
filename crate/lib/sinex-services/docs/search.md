# Search Service

The `SearchService` provides unified event search across multiple dimensions (source, type, time, and text). It serves as the primary discovery mechanism for the `sinex-gateway`.

## API Surface

- `search_events(query: SearchQuery)`
  - Full-text search over JSON payloads.
  - Multi-field filtering (logical AND across sources, types, time).
  - Pagination using `limit` and `offset`.
  - Automatic snippet extraction.

## Implementation Details

### Hybrid Filtering
The search implementation splits filtering between SQL and application logic:
1.  **SQL Layer**: Handles time-range pruning, source/type filtering, and full-text indexing (`ILIKE` or `tsquery`).
2.  **Service Layer**: Handles snippet extraction and highlighting. It extracts a context window (typically 50 chars) around the first match within the JSON payload string.

### Unicode & UTF-8 Safety
All string operations (truncation, context slicing) are performed using Unicode character iterators (`.chars()`). This ensures that search snippets never split a multi-byte character sequence, preventing "corrupted text" bugs.

### Search Scoring
Results are ordered by `ts_coided DESC` by default. When a text query is provided, the database relevance score is utilized to rank the most meaningful results first.

## Safety Constraints

- **Pagination Bounds**: All queries are strictly clamped to a maximum limit (5000) to prevent denial-of-service via massive result sets.
- **Normalization**: Search terms are normalized (trimmed, whitespace-aware) before execution to prevent empty or malicious query strings.
- **HTML Sanitization**: Fragments generated by the database (e.g., `<b>` tags) are stripped or transformed into safe formats for API delivery.
