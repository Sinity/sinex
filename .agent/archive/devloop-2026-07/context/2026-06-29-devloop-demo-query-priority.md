# Demo-Driven Devloop Priority

## Operating Aim

The goal is rapid Sinex capability growth, steered by useful/impressive demos
on real local data. Dogfooding is instrumental: it is useful when it exposes
missing capability, proves a new primitive, or makes the next demo cheaper.

Velocity is part of the product work. If verification, xtask bootstrap, local
schema state, host pressure, duplicate MCP/indexer processes, or stale agent
work block quick iteration, that is not background inconvenience; it is
devloop debt to reduce.

## Current Slice

Two compounding primitives are being moved forward:

- Shared context recall artifact rendering:
  `ContextRecallPackView` now owns the markdown artifact shape so CLI/demos/API/TUI
  can reuse one view primitive instead of growing CLI-private report formats.
- Event query algebra:
  the duplicate flag-shaped `events query` command is being removed, and the
  root `sinexctl query 'events ...'` unit is being widened to cover the useful
  event filters directly: `text`, `provenance`, `time`, `source`,
  `event_type`, `host`, `scope_key`, and `equivalence_key`.

This is intentionally "silo vs algebra" work: useful event search should live
in the descriptor-backed query unit, not in separate CLI flags, completion
mini-languages, report-only structs, or demo-specific glue.

## Demo Rationale

Near-term demos should make Sinex visibly better than a transcript viewer:

- "What was I doing around this agent/session/problem?" should combine chat,
  terminal, source, runtime, event, and machine context.
- "Show me recent meaningful activity" should use one query language over
  event cards, not a separate flag tree.
- "Export/share the recall pack" should be a shared view artifact that can be
  generated by CLI now and reused by API/TUI later.

The query work is not polish. Query is fundamental for every demo because
without expressive, composable retrieval, demos become hand-wired reports.

## Xtask Reliability Incident: 2026-06-29

Observed failure:

- `xtask test -p sinexd -E 'test(desktop_activitywatch_decodes_sqlite_data_json_string)'`
  compiled package-wide sinexd tests instead of the exact production-path
  integration target. Linker fanout then hit memory/IO pressure and died with
  `ld returned 143`, while the reporter collapsed the result to "No tests
  discovered."
- A follow-up `xtask test --list ...` was treated as harmless by the pressure
  guard even though nextest list still compiles test binaries.
- The nextest runner returned on reporter error before `child.wait()`. Dropping
  `std::process::Child` does not kill the cargo/nextest tree, so compiler and
  linker descendants could survive after xtask had already failed.

Patch direction:

- Exact-filter inference now maps nested crate-local integration modules under
  existing root files, so `production_path/desktop.rs` maps to the
  `production_path` binary.
- Check/test pressure gating is warning-only; no more routine
  `--allow-contended-host` bypass ritual. The workload should run when asked,
  with attribution captured.
- Reporter errors before suite discovery include the stderr tail.
- Runner error paths terminate and wait for the nextest process group before
  returning.

Verification:

- Dry run now reports `test_binaries: ["production_path"]` and `db_pool_size: 2`
  for the ActivityWatch regression filter.
- Focused xtask tests passed as job `2000329` / invocation `2003073`:
  `xtask test -p xtask --lib -E 'test(test_infer_test_binaries_maps_nested_module_under_existing_root) | test(no_suite_started_error_includes_stderr_tail)' --bg`.

## RAM Incident: 2026-06-29

Live evidence:

- Before cleanup: about 20 GiB used / 10 GiB available, with 3.3 GiB swap used
  after the worst compile pressure had already subsided.
- The largest resident processes were not Sinex runtime or xtask:
  duplicated Codex MCP stacks dominated the top RSS list.
  The stale Codex tree under PID 8096 owned several Serena MCP servers
  around 0.9-1.0 GiB each, duplicate codebase-memory servers, Polylogue MCPs,
  and TypeScript language servers spawned by Serena.
- The active Codex tree under PID 1869367 owned a Serena MCP whose
  rust-analyzer child had grown to about 2.0 GiB RSS.
- `systemd-cgtop` attributed almost all pressure to the user session
  (`user@1000.service`), not to system Sinex services or nix-daemon.

Actions:

- Terminated the stale Codex tree (`8046`/`8096`). Memory improved to about
  16 GiB used / 15 GiB available.
- Terminated the active Serena-owned `rust-analyzer` child and orphaned MCP
  helpers reparented after the stale tree exited. Memory improved to about
  12 GiB used / 18 GiB available; user-session cgroup settled around 9.6 GiB.
- Left build parallelism/test workload semantics untouched. The issue was
  stale/duplicated agent support processes, not xtask's core workload limit.

Conclusions:

- The actionable optimization is agent-support lifecycle control:
  detect and reap stale Codex/MCP/Serena/codebase-memory/RA trees, especially
  after resume/compaction/reboot-like session changes.
- Serena is expensive in this workspace. Loading it should be deliberate; after
  symbol work, its Rust analyzer child should not be allowed to persist
  indefinitely unless it is actively needed.
- Swap remaining after cleanup is mostly cold GUI/browser/session pages. It is
  not proof of ongoing Sinex pressure.
