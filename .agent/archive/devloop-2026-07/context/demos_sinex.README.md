## Short Position

The strongest near-term Sinex demos should show Sinex as an evidence/context substrate, not as a replacement transcript viewer. Polylogue alone is better for transcript-native operations: listing sessions, reading messages, rendering conversations, provider usage inside AI sessions, and archive-native forensics.

Sinex becomes superior when the question needs evidence Polylogue does not own: terminal commands, git commits and file changes, system activity, source-material provenance, derived work windows, machine signals, and caveated context packs over a time/project/commit.

Therefore the useful cross-stack target is not "import Polylogue into Sinex because import is cool." It is:

```text
Given a time, project, issue, or commit, join Polylogue AI-session evidence with Sinex machine/project evidence and emit a context pack with provenance and caveats.
```

A provisional Polylogue export/import is worth building only if it is a thin versioned bridge that exercises the future Sinex backend contract. It should not create a second bespoke Polylogue implementation inside Sinex.

### 3. `query-algebra-current/`

Current root query algebra demo note.

It answers: "Can Sinex event search be expressed through one descriptor-backed
query unit instead of a separate flag tree or demo-only filter path?"

What it proves:

- `sinexctl query 'events ...'` is now the canonical event search surface.
- The old `sinexctl events query` path has been removed rather than kept as a
  compatibility alias.
- Event filters now include text, provenance, time, source, event_type, host,
  scope_key, and equivalence_key.
- Recent-window syntax now accepts `events since 10m` and lowers it to the same
  descriptor-backed `time >= 10m` predicate; it also combines with `where`.
- Same-field OR and text OR lower where the existing typed API can represent
  them exactly.
- Mixed ORs that would silently widen results are rejected, preserving query
  clarity.

Why it matters:

- Every context/recall demo needs retrieval before rendering.
- Query algebra prevents demos from becoming one-off report code.
- The live dogfood artifact now has a concise query form for "show me recent
  evidence" instead of requiring a flag-shaped context command for every slice.

### 7. `dlq-pressure-live/`

Live operational finding generated on 2026-06-30 from the checkout-local
runtime after `runtime-presence-live/` revealed degraded runtime health.

It answers: "Can Sinex explain why the runtime is degraded instead of only
showing a generic health warning?"

What it proves:

- `sinexctl ops dlq list` gives structured pressure: critical level, throttle
  action, pending messages/bytes, and recommended action.
- `sinexctl ops dlq peek` gives sanitized failure previews.
- `sinexctl ops dlq peek` now also returns grouped explanations in the shared
  `DlqPeekResponse`, so JSON/MCP/table consumers can see the dominant failure
  bucket without bespoke artifact parsing.
- The live backlog is not arbitrary corruption: 20/20 sampled messages are
  `git.commit_d_created` duplicate equivalence-key failures from full-history
  git rescans.
- That points at a concrete event-engine settlement bug: occurrence duplicates
  should suppress/ACK, not become DLQ pressure.

Status:

- Artifact captured.
- Code fix implemented in `JetStreamConsumer::prepare_events` to treat
  `AdmissionDecision::Suppressed` as non-error settlement. `xtask check -p
  sinexd` passed, and after restarting the dev runtime the DLQ count stayed
  stable at 194 over a 30-second sample.
- Grouped DLQ view implemented and verified live: 20 peeked messages collapse
  into one `occurrence_duplicate.equivalence_key_exists` group for
  `dev.events.raw.git.commit_d_created`.
- The same artifact has now grown into a stronger operator-control demo:
  `ops dlq purge` can delete an inclusive stream-sequence range and emit
  structured table/JSON/YAML output. A live cleanup deleted only sequences
  250..259 with `purged_count = 10`, then exposed and fixed the finite-peek
  requirement for deleted stream holes.
- Tail DLQ inspection implemented through the same `DlqPeekRequest` primitive:
  `start_sequence` on the RPC and `sinexctl ops dlq peek --tail` in the CLI.
  Live proof shows default peek still returns oldest sequences 1..5, while tail
  peek returns newest sequences 192..196.
- Explicit coordinate proof also works:
  `ops dlq peek -n 2 --start-sequence 195` returns exactly sequences 195 and
  196.
- Tail proof changed the next diagnosis: newest DLQ entries included
  `empty_material` failures from logical git-history materialization. The
  runtime parser now materializes empty logical-path adapter records as
  descriptor bytes, with focused sinexd test proof and runtime build proof.
- Live restart caveat: DLQ advanced to seq 197, but that material was staged
  before the fix and finalized late on restart; a follow-up tail sample stayed
  at `last_seq = 197`.
- DLQ groups now parse structured JSON error previews into specific buckets:
  the rebuilt live server groups seq 195..197 as
  `error_payload.empty_material` and seq 198..207 as
  `error_payload.slice_arrival_timeout`, turning the newest pressure into a
  concrete stale-material-timeout diagnosis.
- Self-observation priming now appends descriptor JSONL bytes instead of a
  contentless `BEGIN`, and gateway/source-runtime observers prime before use.
  This targets the dominant inspected `slice_arrival_timeout` shape:
  `sinex.self-observation.<module>` materials with no blob or total bytes.
- The descriptor-prime path now also finalizes that descriptor material via the
  shared buffered writer `flush` primitive. Live proof after restart shows
  post-fix self-observation descriptor materials completed with blob-backed
  byte counts, while later telemetry materials can still be active `sensing`
  streams.
- The DLQ tail still advanced to older one-hour `slice_arrival_timeout` rows,
  so the current claim is precise: the new descriptor materials are no longer
  left open; historical/pre-fix material timeouts still need cleanup or paced
  handling.
- Restarting the dev runtime also exposed and fixed an `xtask jobs cancel`
  reliability bug: canceled runtime jobs could be recorded as killed while the
  daemon process kept running.
- `xtask infra status` also stopped misclassifying `xtask test -p sinexd`,
  `cargo`, and `rustc --crate-name sinexd` as dev-local `sinexd`; live status
  now reports only the actual daemon PID/RSS.
- Truncated DLQ previews now preserve and group structured classifiers:
  seq 208 moved from generic `error_payload.unparsed` to
  `error_payload.material_persist_failed` after preview rendering prioritized
  top-level `error`/identity fields and the shared grouping primitive learned
  to extract leading error tokens from truncated JSON previews.

## Recommended Next Build

Tighten and reuse the shared events context recall-pack shape that now exists in `sinex-primitives`.

The work should produce a shared model roughly equivalent to:

- view envelope metadata;
- query echo: anchor/window/project/commit/issue;
- activity groups by source family;
- derived groups;
- self-observation groups;
- salient timeline entries;
- provenance/caveats/coverage;
- stable JSON rendering;
- human markdown rendering.

Then `sinexctl events context`, recall demos, API endpoints, TUI surfaces, and future Polylogue/Sinex joined context should all consume the same algebra instead of growing separate report silos.

Concrete next demo order:

1. Improve recall-pack salience and timeline entries over the existing shared view.
2. Fix ActivityWatch/Hyprland capture quality so multi-source attention reconstruction becomes honest.
3. Add git/file-change family evidence to recall and agent brief.
4. Build agent pre-task briefing as a lens over the same context pack.
5. Add only a thin Polylogue bridge/export once it directly feeds joined context packs and resembles the eventual backend contract.
