# Provenance Migration Guide

## Overview

The Sinex architecture requires that ALL events have provenance. This guide helps migrate existing code that creates events without provenance.

## Step 1: Identify Violations

Search for code that creates events without provenance:

```bash
# Find direct RawEvent::new() calls
rg "RawEvent::new\(" --type rust

# Find Event::new() calls for typed events
rg "Event::new\(" --type rust

# Find events with provenance: None
rg "provenance: None" --type rust
```

## Step 2: Classify Each Event

For each event creation site, determine:

### Is it a First-Order Event?
- Derived from external data (files, network, user input)
- Should reference Source Material captured by sensd
- Needs external provenance (material_id + anchor_byte)

### Is it a Synthesized Event?
- Derived from other events in the system
- Created by automata or processors
- Needs internal provenance (source_event_ids)

## Step 3: Migration Patterns

### Pattern 1: First-Order Events (External Data)

**Before:**
```rust
let event = RawEvent::new("fs-watcher", "file.created", payload);
```

**After:**
```rust
// Option A: Use the new factory method
let event = RawEvent::from_material(
    "fs-watcher",
    "file.created", 
    payload,
    material_id,  // From sensd
    anchor_byte   // Byte offset in material
);

// Option B: Use builder pattern
let event = RawEvent::new("fs-watcher", "file.created", payload)
    .with_provenance(Provenance::from_material(material_id, start, end))
    .with_anchor_byte(Some(anchor_byte));
```

### Pattern 2: Synthesized Events (From Other Events)

**Before:**
```rust
let event = RawEvent::new("canonicalizer", "terminal.command.canonical", payload);
```

**After:**
```rust
// Option A: Use the new factory method
let event = RawEvent::from_events(
    "canonicalizer",
    "terminal.command.canonical",
    payload,
    vec![parent_event_id1, parent_event_id2]
);

// Option B: Use builder pattern
let event = RawEvent::new("canonicalizer", "terminal.command.canonical", payload)
    .with_provenance(Provenance::from_events(parent_ids));
```

### Pattern 3: System Monitoring Events

Events like heartbeats that seem "sourceless" actually need Source Material too.

**Before:**
```rust
// In a monitoring service
let event = RawEvent::new("monitor", "system.heartbeat", json!({
    "uptime": uptime_seconds,
    "memory": memory_usage
}));
```

**After:**
```rust
// Step 1: Register a sensor job with sensd for system metrics
// This happens once at service startup
let job_id = sensd_client.register_sensor_job(
    SensorType::SystemMetrics,
    Duration::from_secs(60)  // Collect every minute
).await?;

// Step 2: Get the current material from sensd
let material = sensd_client.get_current_material(job_id).await?;

// Step 3: Create event with proper provenance
let event = RawEvent::from_material(
    "monitor",
    "system.heartbeat",
    json!({
        "uptime": uptime_seconds,
        "memory": memory_usage
    }),
    material.id,
    material.current_offset
);
```

## Step 4: Update Tests

Tests often create events for testing. Update them to include provenance:

**Before:**
```rust
#[test]
fn test_event_processing() {
    let event = RawEvent::new("test", "test.event", json!({}));
    // ...
}
```

**After:**
```rust
#[test]
fn test_event_processing() {
    // For tests, you can use dummy values
    let material_id = Id::<SourceMaterial>::new();
    let event = RawEvent::from_material(
        "test",
        "test.event",
        json!({}),
        material_id,
        0  // anchor_byte
    );
    // ...
}
```

## Step 5: Add Validation

Add checks to catch violations early:

```rust
// In your event submission code
if event.violates_provenance_requirements() {
    return Err(SinexError::InvalidEvent(
        "Event missing required provenance".to_string()
    ));
}

// Or use debug assertions in development
debug_assert!(
    !event.violates_provenance_requirements(),
    "Event must have provenance"
);
```

## Step 6: Database Enforcement

The database will enforce the XOR constraint, but it's better to catch violations earlier:

```sql
-- This constraint prevents invalid events
ALTER TABLE core.events
ADD CONSTRAINT events_provenance_xor CHECK (
    (material_id IS NOT NULL AND source_event_ids IS NULL)
    OR
    (material_id IS NULL AND source_event_ids IS NOT NULL)
);
```

## Common Pitfalls

### Pitfall 1: "But my event is raw data!"
**Wrong thinking:** "This event represents raw user input, so it doesn't need provenance."

**Correct thinking:** Even "raw" observations must be captured as Source Material first. The event is an *interpretation* of that material.

### Pitfall 2: "System events are special"
**Wrong thinking:** "Heartbeat events are generated by the system, not from external sources."

**Correct thinking:** System state is also external data that must be captured. Register a sensor with sensd for system metrics.

### Pitfall 3: "Tests don't need real provenance"
**Wrong thinking:** "It's just a test, I'll use None for provenance."

**Correct thinking:** Tests should model real behavior. Use dummy but valid provenance values.

## Verification

After migration, verify no violations remain:

```rust
// Add this to your CI pipeline
#[test]
fn no_provenance_violations() {
    // Query database for events without provenance
    let violations = sqlx::query!(
        "SELECT COUNT(*) as count FROM core.events 
         WHERE material_id IS NULL AND source_event_ids IS NULL"
    )
    .fetch_one(&pool)
    .await
    .unwrap();
    
    assert_eq!(violations.count, 0, "Found events without provenance!");
}
```

## Summary

1. ALL events need provenance - no exceptions
2. First-order events need Source Material references
3. Synthesized events need parent event references
4. Use the new factory methods (`from_material`, `from_events`) to ensure correctness
5. The database will enforce this, but catch violations early in code
