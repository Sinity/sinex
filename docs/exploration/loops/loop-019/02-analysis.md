# Loop 019 - Dynamic Event Emission for Registry-only Schemas

Scope
- Dynamic event creation paths (`Event::dynamic`, `test_event`, JSON builders).
- Registry-only sources: `shell.*`, `terminal.kitty`, `atuin`, `journald.satellite.heartbeat`, `system.*_historical`.

Dynamic Emission Search
- No production usage of `Event::dynamic` found in the codebase.
- `Event::test_event` and other JSON-based helpers appear primarily in tests.
- Production nodes generally build typed payloads and serialize them to JSON before emitting.

Typed Payload Coverage for Shell/Terminal
- `crate/lib/sinex-core/src/types/events/payloads/shell.rs` defines typed payloads via `define_event_payload!` for:
  - `shell.kitty/command.executed`, `shell.kitty/command.completed`.
  - `terminal.kitty/session.started`, `terminal.kitty/session.ended`.
  - `shell.atuin/*`, `shell.history/command.imported`, `shell.bash_histfile/*`, `shell.zsh_histfile/*`, `shell.fish_history/command.historical`.
  - `atuin/entry.imported` and `shell/command.imported`.
- These are typed, but earlier inventory scripts missed them because they are generated by a macro rather than direct `#[event_payload]` annotations.

Registry-only Entries Without Code Emitters
- `journald/satellite.heartbeat` and `system/*_historical` have JSON schemas in `schemas/v1` but no `EventPayload` definitions or emitters found in code.
- Searches for these event types yielded only test regression artifacts, not production emitters.

Findings
- Registry-only entries for shell/terminal/atuin are actually backed by typed payloads in `shell.rs`; the mismatch was due to inventory extraction that ignored the macro.
- The remaining registry-only entries (`journald.satellite.heartbeat`, `system.*_historical`) appear to be legacy or external-only schemas not emitted by current code.
- Dynamic event emission does not explain registry-only schemas; most emitters use typed payloads or explicit `EventBuilder` with typed payload JSON.

Risks
- Legacy schemas that are not emitted can remain in the registry indefinitely, increasing drift and confusion.
- Inventory tooling that ignores macro-based payloads will continue to mis-report schema drift.

Opportunities
- Enhance schema inventory tooling to parse `define_event_payload!` invocations.
- Clarify in docs which schemas are legacy/external-only to avoid confusion.
- Consider pruning registry-only schemas with no emitters if they are deprecated.
