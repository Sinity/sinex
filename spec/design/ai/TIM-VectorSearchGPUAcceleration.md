# TIM-VectorSearchGPUAcceleration: GPU Acceleration for Large-Scale Vector Search

*   **Relevant ADR:** `[ADR-007-LargeScaleVectorSearchStrategy.md](docs/adr/ADR-007-LargeScaleVectorSearchStrategy.md)` (GPU DB as future scale option)
*   **Original UG Context:** Section 17.5
*   **Vision Document Reference:** Part IV.D.4 (Future Phase - from `openai_dr_sinex_5.md` structure, not current vision doc) - *Note: This reference might be outdated; the current Vision Doc may not have this specific section. The primary driver is ADR-007.*

This TIM details strategies for GPU-accelerating vector search when Exocortex datasets grow very large (e.g., >10-50 million vectors), necessitating migration from CPU-bound `pgvector` to a dedicated GPU-enabled vector database like Milvus or Qdrant.

## 1. Rationale Summary (from ADR-007)

While `pgvector` on CPU is performant for many Exocortex scales, GPU acceleration becomes necessary for very large vector counts or extreme QPS/latency requirements. This involves using an external vector database.

## 2. External Vector Databases with GPU Support [UG Sec 17.5.2]

### 2.2. Qdrant [CR4]

*   Vector database in Rust.
*   Supports GPU for HNSW indexing/search, often with quantization (e.g., float16 on GPU).

## 3. Hybrid Deployment Model (PostgreSQL + External Vector DB) [UG Sec 17.5.3, CR4]

*   **Architecture:**
    1.  PostgreSQL (`sinex_db`): Stores all metadata (`core_artifacts`, etc.), FTS indexes. Remains "source of truth" for original textual data and potentially a CPU-accessible copy of embeddings.
    2.  Embeddings are generated by `EmbeddingAgent`.
    3.  Embeddings are synchronized from PostgreSQL to the external GPU vector DB.
*   **Hybrid Query Flow:**
    1.  Metadata/FTS query parts -> PostgreSQL.
    2.  Vector similarity search -> External GPU vector DB.
    3.  Results combined/re-ranked in application layer (e.g., with RRF).

## 4. Data Synchronization Mechanisms [UG Sec 17.5.3, CR4, CR5]

*   **Debezium CDC (Change Data Capture) via Kafka [CR5]:**
    *   Debezium PostgreSQL connector captures row changes from `artifact_embeddings` (or source table).
    *   Changes published to Kafka.
    *   Kafka consumer app writes embeddings to Milvus/Qdrant. (Near real-time).
*   **Event-Driven Updates / Dual Writes [CR4]:**
    *   `EmbeddingAgent` (or service handling new content) writes embeddings to both PostgreSQL *and* external vector DB.
    *   Handle distributed transaction or eventual consistency:
        *   Write to PG first. If success, try vector DB. If vector DB fails, enqueue for retry.
*   **Batch Synchronization:** Periodically query PG for new/updated embeddings and bulk-insert to external vector DB. Simpler, adds sync latency.

## 5. Cost-Effectiveness and Cloud GPU Instance Selection [UG Sec 17.5.4, CR4]

*   **Crossover Point for GPU:** Becomes cost-effective for >10-50 million vectors.
*   **Cost Reduction Example [CR4]:** ~63% cost reduction at 100M vectors with GPU vs. scaled CPU.
*   **Cloud GPU Instances (AWS Examples):**
    *   Medium Workloads (up to 50-100M vectors): `g5.xlarge` (NVIDIA A10G).
    *   Large-Scale: `p3.2xlarge` (V100) or newer `p4d/p5` (A100/H100).
*   **Spot Instances:** For fault-tolerant workloads like batch index building or query nodes with failover.

## 6. Memory and Storage Requirements for GPU Setups [UG Sec 17.5.5, CR4]

*   **GPU VRAM (Primary Bottleneck):**
    *   Index (or hot parts + quantization codes) should fit in VRAM.
    *   Rule of thumb: VRAM needed ~2x-4x raw vector data size (depends on index type, quantization).
    *   Example: 100M 768-dim FP32 vectors = ~307GB raw. Requires multiple high-VRAM GPUs or aggressive quantization (FP16/INT8).
*   **System RAM (Host):** Still crucial for OS, DB components, data staging.
*   **Storage (Disk):** Fast NVMe SSDs for full index on disk (if paged to VRAM) and raw vectors/metadata.

## 7. Migration Strategy: `pgvector` to Milvus/Qdrant [UG Sec 17.5.6, CR5]

1.  **Setup Target Vector DB:** Deploy Milvus/Qdrant with GPU. Define collections/indexes.
2.  **Initial Bulk Load:**
    *   Export embeddings (vector + ID) from PostgreSQL `artifact_embeddings`/`event_embeddings` (e.g., to Parquet/JSONL).
    *   Use target DB's bulk import tools/SDKs. Build initial GPU indexes.
3.  **Implement Delta Synchronization (CDC Preferred):** Debezium/Kafka, or event-driven dual writes, or batch sync.
4.  **Query Routing & Cutover:**
    *   Modify Exocortex search service to query either `pgvector` or external DB.
    *   Feature flag or routing logic to gradually shift traffic. Monitor performance.
5.  **Validation:** Compare search results for consistency.
6.  **Full Cutover:** Route all semantic search to GPU DB.
7.  **Fallback/Rollback Plan:** Ability to switch back to `pgvector`. Ensure `pgvector` can be updated with changes made while external DB was primary.
