# TIM-PKMContentCRDT_Yjs: PKM Note Content with Yjs CRDTs

*   **Relevant ADR:** `[ADR-004-PKMNoteContentManagementAndSync.md](docs/adr/ADR-004-PKMNoteContentManagementAndSync.md)` (Decision: DB-Native with Yjs)
*   **Original UG Context:** Section 13.2 (heavily influenced by ADR-004)

This TIM details the implementation for managing Personal Knowledge Management (PKM) note content using Yjs Conflict-Free Replicated Data Types (CRDTs), with the Exocortex database as the canonical store. This approach largely supersedes file-based bi-directional sync and textual merging for the primary editing workflow.

## 1. Rationale Summary (from ADR-004)

Yjs CRDTs for PKM note content provide robust, conflict-free versioning and merging, ideal for a database-native content strategy. This ensures data integrity, optimizes for agentic integration, and provides a foundation for future collaborative or multi-device editing scenarios. The database (`core_artifacts`, `core_artifact_contents`, and a Yjs delta store) is the single source of truth.

## 2. Core Data Model and Workflow

*   **`core_artifacts`:** An entry with `artifact_type='pkm_note'` represents a conceptual PKM note. Its `artifact_id` is stable. `canonical_identifier` might be a user-chosen unique title or path fragment.
*   **`core_artifact_contents`:** Stores periodic, full snapshots of the note's content, typically rendered as Markdown from the Yjs document state. `core_artifacts.current_content_id` points to the latest snapshot's `content_id`.
*   **`core.pkm_note_yjs_deltas` (New Table for Yjs Updates):** This table stores the immutable sequence of Yjs update blobs for each note.
    ```sql
    CREATE TABLE IF NOT EXISTS core.pkm_note_yjs_deltas (
        delta_id                ULID PRIMARY KEY DEFAULT gen_ulid(), -- Unique ID for the delta itself
        artifact_id             ULID NOT NULL REFERENCES core.artifacts(artifact_id) ON DELETE CASCADE, -- Links to the PKM note artifact
        -- For strict ordering within an artifact_id, if needed beyond ULID time:
        -- delta_sequence_num   BIGSERIAL NOT NULL, -- Or a Lamport/HLC timestamp
        yjs_update_blob         BYTEA NOT NULL,    -- The binary Yjs update data
        originator_actor        TEXT NOT NULL,     -- e.g., 'user_neovim_plugin_instance_X', 'agent_AutoLinker_v1'
        ts_created              TIMESTAMPTZ NOT NULL DEFAULT now(),
        -- Optional: Store Yjs state vector *before* this delta was applied, for specific sync protocols
        -- yjs_prev_state_vector BYTEA NULLABLE,
        UNIQUE (artifact_id, ts_created, delta_id) -- Ensure unique deltas, ts_created helps ordering if ULIDs are too close
    );
    CREATE INDEX IF NOT EXISTS idx_pkm_note_yjs_deltas_artifact_ts ON core.pkm_note_yjs_deltas (artifact_id, ts_created ASC, delta_id ASC);
    ```
*   **Editing Workflow (e.g., Neovim `sinnix-nvim` plugin):**
    1.  **Load Note:** Plugin requests note content for `artifact_id` from Exocortex backend.
        *   Backend retrieves the latest Markdown snapshot from `core_artifact_contents`.
        *   It also retrieves all Yjs update blobs from `core.pkm_note_yjs_deltas` for that `artifact_id` *since* the timestamp or sequence number associated with that snapshot.
        *   Backend (or plugin) initializes a local Yjs document from the snapshot (if snapshot is a Yjs state vector) or by parsing the Markdown into Yjs, then applies subsequent update blobs.
    2.  **Local Edits:** User edits in Neovim. Plugin translates text changes into Yjs operations on its local Yjs document replica.
    3.  **Save Note:** Plugin sends new Yjs update blobs (generated by `Yjs.encodeStateAsUpdate(localYDoc, optionalRemoteStateVector)` or similar, representing changes since last sync) to the Exocortex backend.
    4.  **Backend Processing:**
        a.  Receives Yjs update blob(s) for an `artifact_id`.
        b.  Applies them to its canonical Yjs document instance for that note (potentially fetching its current state vector first).
        c.  Persists the new Yjs update blob(s) as new row(s) in `core.pkm_note_yjs_deltas`.
        d.  Periodically (or on significant change, or after N deltas), the backend generates a new full Markdown snapshot from the updated Yjs document state. This new snapshot is saved as a new row in `core_artifact_contents`, and `core_artifacts.current_content_id` is updated.
        e.  A `sinex.pkm.note_updated_yjs` event is logged, referencing the `artifact_id` and the new `delta_id`(s).

## 3. Yjs Implementation Details

*   **Yjs Libraries:**
    *   **Backend (Rust):** Use Rust bindings for Yrs (the Rust port of Yjs), e.g., `yrs` crate.
    *   **Neovim Plugin (Lua):** May require:
        *   Lua bindings for Yrs if available and performant.
        *   Or, an embedded JavaScript engine (like `mini.nvim` sometimes uses for JS tasks) to run `yjs` directly.
        *   Or, the Neovim plugin communicates via Msgpack-RPC to a small local helper process (Node.js or Deno) that manages the Yjs document and exposes a simple API for applying text diffs/ops and getting Yjs updates.
*   **Yjs Update Blobs (`BYTEA`):** These are compact binary representations of changes.
*   **Yjs State Vectors (`BYTEA`):** Compact representation of the state of all origins in a Yjs document, used to calculate minimal update blobs.
*   **Compression for `yjs_update_blob` (UG Sec 13.2.4, CR3):** PostgreSQL's TOAST mechanism will automatically compress `BYTEA` columns using `pglz` (or `lz4` if default changed, `zstd` if compiled in PG15+). The "ZSON for Yjs" mention in CR3 likely refers to the effectiveness of general-purpose binary compression on Yjs's efficient binary format, rather than ZSON itself directly processing Yjs blobs.
*   **Snapshotting and Garbage Collection (Compaction) [UG Sec 13.2.3, OR3, CR3]:**
    *   **Snapshotting:** To avoid replaying an excessive number of deltas from `core.pkm_note_yjs_deltas` when loading a note:
        1.  Periodically (e.g., after N deltas, or if total delta size > threshold, or daily):
            a.  Load the full Yjs document state by applying all deltas for an `artifact_id`.
            b.  Generate a new full snapshot:
                *   Either as a Yjs state update that represents the entire document: `Yjs.encodeStateAsUpdate(ydoc)`. Store this special "snapshot delta" in `core.pkm_note_yjs_deltas` with a flag, AND/OR
                *   Render to Markdown and store in `core_artifact_contents`.
            c.  Record the `delta_id` or `ts_created` of the last delta incorporated into this snapshot.
    *   **Garbage Collection/Pruning Deltas:** Older individual delta blobs from `core.pkm_note_yjs_deltas` that are now covered by a recent full snapshot (either Yjs state update or a Markdown snapshot in `core_artifact_contents` whose generation time is known) can be moved to an archive table or deleted if very old, to keep the "hot" delta table smaller. This depends on how far back version history needs to be "live" vs. reconstructable from older full snapshots + subsequent deltas.

## 4. Markdown Stable Heading ID Generation [UG Sec 13.2.5, CR3]

To create persistent anchor links within Markdown content that survive minor edits.

*   **Algorithm (from UG Conceptual TypeScript example):**
    1.  **Parse Markdown (from Yjs-rendered snapshot):** Identify headings (text, level).
    2.  **Hierarchical Numbering:** Assign `1`, `1.1`, `1.1.2`, etc., based on level and sequence.
    3.  **Content Hashing:** For each heading's normalized text (lowercase, punctuation removed, whitespace normalized), compute a short crypto hash (e.g., first 8-12 chars of SHA-256/BLAKE3 hex).
    4.  **Stable ID:** Combine: `<hierarchical_number>-<short_content_hash>` (e.g., `1.2.3-a1b2c3d4`).
*   **Storage/Usage:**
    *   These stable IDs can be stored as metadata with the Yjs representation of headings (if Yjs custom types or attributes are used).
    *   When rendering Markdown from Yjs, these IDs are injected as HTML `id` attributes (e.g., `<h2 id="1.2.3-a1b2c3d4">...</h2>`).
    *   Used in Wikilinks: `[[NoteName#1.2.3-a1b2c3d4]]`. The `core_artifact_links` resolution logic would need to understand these.
*   **Implementation:** This logic would reside in the Exocortex backend (when generating Markdown snapshots or processing links) or potentially within the `sinnix-nvim` plugin if it needs to generate/resolve these IDs client-side for navigation.

## 5. Impact on Filesystem Sync and Merging

As per ADR-004, bi-directional filesystem sync and textual 3-way merging (UG Sec 13.1, 13.3) are **not** the primary mechanism for PKM notes managed by Exocortex.
*   **Filesystem Vault as Export/Cache:** The user's local Markdown file vault can be populated by an Exocortex agent that:
    1.  Queries `core_artifacts` for `pkm_note` types.
    2.  For each note, gets the latest Markdown snapshot from `core_artifact_contents` (derived from Yjs).
    3.  Writes this Markdown to a file in the vault (e.g., path based on `core_artifacts.canonical_identifier`).
    This is a one-way sync from DB to filesystem for consumption by other tools or for user browsing.
*   **Handling External Edits to Files:** If a user edits these exported Markdown files directly (outside the Yjs-aware `sinnix-nvim` flow):
    1.  The Filesystem Ingestor (`TIM-FilesystemIngestionLogic.md`) detects the change.
    2.  An agent compares the modified file content with the canonical Yjs-derived Markdown in `core_artifact_contents`.
    3.  If different, it can:
        a.  Flag a conflict: `sinex.pkm.external_modification_conflict` event. User resolves manually (e.g., copy-paste changes into Neovim to get them into Yjs, or choose to overwrite DB version with file version).
        b.  Attempt to apply the textual diff from the file as a new set of operations to the Yjs document (complex, might lead to merge issues if not careful, as it bypasses Yjs's awareness of concurrent changes).
        c.  Default: Treat the file edit as a "fork" and import its content as a *new version proposal* to the Yjs document, letting Yjs merge semantics (or user via diff view) handle it.

This Yjs-centric model prioritizes data integrity and robust merging for the canonical Exocortex representation of PKM notes.
