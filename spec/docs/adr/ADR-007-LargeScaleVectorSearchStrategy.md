# ADR-007: Large-Scale Vector Search Strategy

*   **Status:** Implemented
*   **Date:** 2024-03-11
*   **Implementation Date:** 2025-07-17
*   **Context & Problem Statement:**
    Semantic search using vector embeddings is a core capability of the Exocortex. As the volume of embeddable content (PKM notes, web archives, event payloads) grows potentially into millions or tens of millions of vectors, the performance (latency, QPS) and scalability of the vector search solution become critical. The default approach is to use `pgvector` within PostgreSQL, which is CPU-bound. For very large scale, GPU-accelerated dedicated vector databases are an option.

*   **Discussed Options:**

    1.  **`pgvector` on CPU (Baseline):**
        *   **Description:** Use `pgvector` extension in PostgreSQL with HNSW or IVFFlat indexes. All indexing and search operations run on the CPU cores available to the PostgreSQL server.
        *   **Pros:**
            *   **Simplicity:** Integrated directly into the primary Exocortex datastore. No additional services to deploy or manage.
            *   **Unified Data Management:** Embeddings live alongside their source metadata (`core_artifact_contents`, `core.events`), simplifying queries that combine vector search with metadata filtering.
            *   **Cost-Effective at Small to Medium Scale:** Leverages existing PostgreSQL hardware. No separate licensing or infrastructure costs for a dedicated vector DB.
            *   **Good Performance for Many Use Cases:** `pgvector` with HNSW can provide excellent performance (e.g., ~1800 QPS at 91% recall on 50M vectors in one cited test [CR4], sub-100ms p99 latency for 1M document hybrid search [CR3]) on commodity CPU hardware for typical Exocortex scales.
        *   **Cons:**
            *   **CPU-Bound Limits:** Performance will eventually plateau or degrade as vector dataset size and query volume exceed CPU capacity. Cannot leverage GPU acceleration.
            *   **Resource Contention:** Vector search operations compete for CPU resources with other PostgreSQL operations (ingestion, standard queries, agent processing).
            *   **Scalability Ceiling:** For extremely large datasets (e.g., >50-100 million high-dimensional vectors) or very high QPS requirements, a CPU-only solution might not be sufficient or cost-effective compared to specialized GPU solutions.

    2.  **External GPU-Accelerated Vector Database (e.g., Milvus, Qdrant):**
        *   **Description:** Deploy a dedicated vector database service (like Milvus or Qdrant) that supports GPU-accelerated indexing (e.g., Milvus CAGRA, GPU_IVF_PQ) and search. Embeddings are generated by Exocortex agents and synchronized from PostgreSQL (or written dually) to this external database. Hybrid queries involve querying PostgreSQL for metadata/FTS and the vector DB for ANN search, then combining results in the application layer.
        *   **Pros:**
            *   **Massive Scalability & Performance:** GPUs can provide orders-of-magnitude speedup for ANN search on very large datasets (e.g., 50x faster [CR4]). Can handle hundreds of millions to billions of vectors with low latency and high QPS.
            *   **Specialized Features:** These databases often offer advanced features like vector quantization on GPU, filtering during ANN search, tunable consistency levels, and optimized data sharding/replication.
            *   **Offloads Vector Search from PostgreSQL:** Frees up PostgreSQL CPU resources for other tasks.
        *   **Cons:**
            *   **Increased Complexity:** Adds another major service to deploy, manage, monitor, and back up. Requires expertise in that specific vector DB.
            *   **Data Synchronization:** Requires a robust mechanism (e.g., Debezium CDC via Kafka [CR5], dual writes with retry queues, batch sync) to keep embeddings in the external DB consistent with the source data in PostgreSQL. Adds latency and potential consistency challenges.
            *   **Hardware Costs:** Requires GPU hardware (either local or cloud instances), which can be expensive.
            *   **Hybrid Query Complexity:** Application logic for hybrid search becomes more complex, needing to query two systems and merge results.
            *   **Network Overhead:** Communication between Exocortex application layer and the external vector DB adds network latency.

    3.  **`pgvector` with Future GPU Support (Hypothetical/Emerging):**
        *   **Description:** This option anticipates potential future developments where `pgvector` itself (or a related PostgreSQL extension/architecture) might gain direct support for GPU offloading of ANN search operations, possibly through mechanisms like Foreign Data Wrappers (FDWs) to GPU-accelerated libraries or direct integration with GPU compute frameworks.
        *   **Pros:** Would offer the "best of both worlds"—integrated experience of `pgvector` with the performance of GPU acceleration.
        *   **Cons:** Largely speculative at present. Depends on external open-source or commercial developments. Timeline and feasibility uncertain. Not a basis for current architectural decisions requiring GPU scale.

*   **Decision:**
    The Exocortex will adopt a **phased approach, starting with `pgvector` on CPU (Option 1) and planning for a potential future migration to an External GPU-Accelerated Vector Database (Option 2)** if scale or performance demands necessitate it.
    1.  **Initial Implementation (Current & Near-Term):** All vector embeddings will be stored and indexed within PostgreSQL using `pgvector` (with HNSW as the default index type per ADR-005). All semantic search queries will operate against `pgvector` on the CPU.
    2.  **Scalability Trigger & Migration Path:**
        *   Performance of `pgvector` will be continuously monitored (latency, QPS, recall under load) as the Exocortex dataset grows.
        *   If vector search performance becomes a bottleneck for user experience or agent operations (e.g., consistently exceeding latency SLOs for datasets >10-50 million vectors, or if CPU load from `pgvector` significantly impacts other critical database operations), a migration to a dedicated GPU-accelerated vector database (e.g., Milvus or Qdrant) will be initiated.
        *   The architecture will be designed to facilitate such a migration:
            *   Semantic search logic in Exocortex agents/services will be encapsulated to allow switching the backend vector search provider.
            *   A data synchronization pipeline (initially batch, potentially evolving to CDC with Debezium/Kafka as per CR5) will be developed to populate the external vector DB from `pgvector` (which remains the "source of truth" for embeddings before full cutover).
            *   The hybrid query model (PostgreSQL for metadata/FTS, external DB for vector search, application-layer merge with RRF) will be adopted during and after migration.

*   **Rationale for Decision:**
    1.  **Pragmatism and Initial Simplicity:** `pgvector` on CPU is sufficient and significantly simpler to deploy and manage for the initial and medium scales envisioned for a personal Exocortex. It avoids premature introduction of complex external dependencies and GPU hardware requirements.
    2.  **Proven Performance at Moderate Scale:** Benchmarks (CR3, CR4) show that `pgvector` can handle millions to tens of millions of vectors effectively on commodity CPUs, meeting the performance needs of many Exocortex use cases.
    3.  **Cost-Effectiveness (Initial):** No additional infrastructure or licensing costs beyond PostgreSQL and existing hardware.
    4.  **Clear Upgrade Path:** The decision explicitly plans for a future where GPU acceleration might be needed. By anticipating a hybrid architecture and data synchronization, the migration path is made clearer.
    5.  **Data Integrity:** Keeping `pgvector` as the initial "source of truth" for embeddings simplifies data management and backup until a full migration to an external system is complete and validated.

*   **Consequences:**
    *   Vector search performance will be limited by CPU capabilities in the initial deployment.
    *   The Exocortex team (or Claude) will need to implement robust performance monitoring for `pgvector`.
    *   If migration becomes necessary, there will be a significant development effort to deploy and integrate the chosen external vector database, set up data synchronization, and adapt query logic.
    *   NixOS configurations for deploying Milvus/Qdrant (potentially in Docker with GPU passthrough, as per UG Sec 17.5.2) will be needed for the migration phase.
