# Sinex Configuration: Direct Approach

**Status**: ✅ Implemented - Ready for testing  
**Date**: 2025-06-27  
**Impact**: Simple, clear configuration without over-engineering

## Problem with Previous Approach

The initial configuration simplification was still over-engineered:
- **Preset system**: Complex abstraction that users don't need
- **TOML handling**: Unnecessary when NixOS can generate config directly
- **Environment variables**: Added complexity without value
- **Retention limits**: Artificial constraints when data volumes are minimal
- **Complex migration**: Over-engineered transition system

## Direct Configuration Philosophy

### Core Principles
1. **NixOS-first**: Configuration generated directly from NixOS options
2. **Required settings explicit**: No dangerous defaults (git-annex path must be set)
3. **Full-featured defaults**: Enable comprehensive capture by default
4. **Simple on/off switches**: No complex presets or abstraction layers
5. **Command-line overrides**: Direct flags for common adjustments

### Configuration Structure

```nix
services.sinex = {
  enable = true;
  
  # REQUIRED: Must be explicitly set (no defaults to avoid mistakes)
  targetUser = "sinity";
  annexRepo = "/home/sinity/sinex-data";
  
  # Event sources: full-featured defaults, easy to disable
  eventSources = {
    filesystem = true;          # File changes
    terminal = true;            # Shell commands
    windowManager = true;       # Window focus/workspaces
    clipboard = true;           # Clipboard changes
    systemEvents = true;        # D-Bus, journal, system
    
    # Advanced (disabled by default)
    processMonitoring = false;  # All processes (needs root)
    networkMonitoring = false;  # Network (needs root) 
    screenCapture = false;      # Screenshots (privacy sensitive)
  };
  
  # Observability: simple on/off (on = full stack)
  observability.enable = true;  # Prometheus + Grafana + dashboards
  
  # Storage: infinite retention by default (data is minimal)
  storage.dataRetention = null; # No limits
};
```

## Implementation Details

### NixOS Module (`nixos/modules/sinex-config.nix`)
- **Direct option mapping**: No abstraction layers
- **Required field validation**: Fails build if git-annex not configured
- **Full-featured defaults**: Comprehensive capture enabled by default
- **Simple observability**: Two modes: off (basic) or on (full stack)

### Collector Binary Updates
- **NixOS config support**: Loads TOML generated by NixOS module
- **Command-line overrides**: Direct flags for common adjustments
- **Legacy compatibility**: Falls back to old config format if needed
- **Required validation**: Enforces git-annex path requirement

### Configuration Flow
1. **NixOS generates TOML**: Module creates config file from options
2. **Collector loads config**: Binary reads generated configuration
3. **Auto-discovery**: Smart defaults for tool paths and settings
4. **Validation**: Ensures required settings and git-annex repository

## Data Retention Strategy

### Why Infinite Retention Makes Sense

1. **Minimal Data Volumes**
   - Even heavy usage: <1GB/year compressed
   - Storage cost: <$3/year for 10 years
   - Modern storage: Trivial impact

2. **TimescaleDB Optimization**
   - ULID-based partitioning for efficient time queries
   - Automatic compression for old data
   - Materialized views for aggregated metrics
   - Query performance maintained over time

3. **Value Maximization**
   - Complete digital activity history
   - Long-term pattern analysis
   - Research and insight potential
   - Personal data archaeology

### Storage Management
- **Git-annex for large files**: >10MB automatically stored externally
- **TimescaleDB compression**: 70-90% reduction for time-series data
- **Deduplication**: Automatic for repeated content
- **Tiered storage**: Future option for very old data

## Command-Line Interface

### Basic Usage
```bash
# NixOS-generated config (typical)
sinex-collector --config /etc/sinex/collector.toml

# Command-line configuration  
sinex-collector --annex-repo /home/user/sinex-data

# Disable specific sources
sinex-collector --annex-repo /path --disable filesystem,clipboard

# Validate configuration
sinex-collector --config /path/to/config.toml --validate-config
```

### Development/Testing
```bash
# Dry run mode
sinex-collector --annex-repo /path --dry-run

# File logging only
sinex-collector --annex-repo /path --no-db --event-log events.json

# Verbose output
sinex-collector --annex-repo /path --verbose
```

## Git-Annex Requirement

### Why Required
- **Large file storage**: Screenshots, terminal recordings, large clipboard content
- **Deduplication**: Automatic via content hashing
- **Efficiency**: Keeps database lightweight
- **Integrity**: Content-addressed storage ensures data integrity

### Setup
```bash
# Initialize git-annex repository
mkdir /home/user/sinex-data
cd /home/user/sinex-data
git init
git annex init "sinex-data-$(hostname)"

# Configure in NixOS
services.sinex.annexRepo = "/home/user/sinex-data";
```

### Validation
The system validates git-annex repository on startup:
- Path exists
- Is a git repository  
- Has git-annex initialized (`.git/annex/` directory exists)
- Fails fast if any requirement not met

## Observability Design

### Two-Mode System
- **Disabled**: Basic health checks, warn-level logging only
- **Enabled**: Full stack - Prometheus + Grafana + 7 dashboards + alerts

### When Enabled (Default)
- **Prometheus**: Metrics collection with infinite retention
- **Grafana**: Web interface on port 3000 with auto-provisioned dashboards
- **Exporters**: Node exporter + PostgreSQL exporter
- **Dashboards**: 7 pre-built dashboards covering all aspects
- **Alerts**: System health, resource usage, service status

### Dashboard Suite
1. **System Overview**: High-level system health and activity
2. **Event Analysis**: Event patterns and source analysis  
3. **Pipeline Performance**: Event processing and worker metrics
4. **System Health**: Resource usage and service status
5. **Worker Performance**: Concurrent processing and queue metrics
6. **Continuous Aggregates**: TimescaleDB metrics and trends
7. **Troubleshooting**: Error analysis and debugging tools

## Migration from Complex System

### For Existing Users
1. **Remove preset configuration**: Delete complex TOML files
2. **Configure NixOS module**: Set required options (user, annexRepo)
3. **Enable observability**: Set `observability.enable = true`
4. **Deploy**: `nixos-rebuild switch`

### Backward Compatibility
- Legacy TOML files still work as fallback
- Command-line interface unchanged for basic usage
- Gradual migration supported
