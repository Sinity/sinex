## 1. Introduction & Philosophy

### 1.1. Purpose of the Data Substrate in Exocortex

The Data Substrate is the bedrock upon which the entire Sinnix Exocortex is built. Its primary purpose is to provide a robust, scalable, and universally accessible persistence layer for all data captured and generated by the system. This includes raw telemetry from ingestors, processed and structured information, user-created knowledge artifacts, large binary objects, and the system's own operational metadata. A well-architected data substrate is essential for achieving the Exocortex's goals of comprehensive capture, emergent understanding, and long-term cognitive augmentation.

### 1.2. Core Principles Guiding Data Substrate Design

The design of the Exocortex Data Substrate is guided by several core principles, largely derived from the Vision Document (Part III.1.1 and I.3):

*   **Universal Log & Append-Only Truth (`raw.events`):** All incoming data first lands in the `raw.events` table as an immutable, append-only log. This preserves the original fidelity of captured information and serves as the ultimate source of truth.
*   **Auditability & Replayability:** Every piece of structured knowledge or automaton decision must be traceable back to its originating raw events through provenance chains. This enables reprocessing (replay) of historical data when automaton logic improves or requires correction.
*   **Emergent Structure:** Rigid schemas are not imposed prematurely at the point of ingestion. Higher-level organization (typed domain tables, knowledge graph connections, semantic tags) is created *downstream* from `raw.events` through promotion and enrichment pipelines.
*   **Lossless Transformation:** Structuring processes are additive and traceable. Derived data never overwrites the source truth.
*   **Data Integrity:** Mechanisms for schema validation, content hashing, and referential integrity are employed to ensure data consistency and correctness.
*   **Performance & Scalability:** The substrate must handle high volumes of event data and support efficient querying and analysis as the Exocortex grows over a lifetime of use.
*   **Local-First & Data Sovereignty:** The primary data store is local to the user, ensuring control and ownership.

### 1.3. Overview of Key Technologies Chosen

The Exocortex data substrate relies on a carefully selected stack of technologies:

*   **PostgreSQL (15+):** The primary relational database management system. Chosen for its maturity, ACID compliance, powerful SQL dialect, rich support for JSONB, and extensive ecosystem of extensions. It serves as the central hub for structured metadata, event logs, and textual content.
*   **TimescaleDB:** A PostgreSQL extension that transforms standard tables (like `raw.events`) into hypertables, automatically partitioned by time. Crucial for managing large volumes of time-series event data.
*   **`pgx_ulid`:** A PostgreSQL extension providing a native `ulid` data type and generator functions. ULIDs are used as primary keys for time-ordered, globally unique identification.
*   **`pgvector`:** A PostgreSQL extension for storing and searching vector embeddings, enabling semantic search capabilities.
*   **`pg_jsonschema`:** A PostgreSQL extension for validating JSONB data against JSON Schema definitions directly within the database.
*   **`git-annex`:** A content-addressed filesystem for managing large binary objects (blobs), integrated with PostgreSQL for metadata storage.

## 2. The Canonical Event Substrate: `raw.events`

> **✅ IMPLEMENTATION STATUS: COMPLETE** - Table created, indexed, TimescaleDB hypertable operational, ULID primary keys working

At the absolute core of the Exocortex lies the `raw.events` table, serving as the universal, immutable entry point for all data.

### 2.1. Architectural Role of `raw.events`

The `raw.events` table is the append-only "source of truth" for the Exocortex. Every piece of information, from a keystroke to a complex analytical result generated by an automaton, is first recorded here as an event. This design ensures:
*   **Complete History:** No raw data is ever lost or modified in place.
*   **Decoupling:** Ingestion processes can write to `raw.events` without needing to know about all downstream consumers or complex schemas.
*   **Resilience:** If downstream processing fails or needs correction, the original raw event is always available for reprocessing.

### 2.2. High-Level Schema Overview of `raw.events`

The `raw.events` table is designed to capture essential metadata while keeping the event-specific data flexible within a JSONB payload.
*   **Key Columns:**
    *   `id ULID PRIMARY KEY`: The globally unique, time-sortable identifier for the event.
    *   `source TEXT`: Canonical identifier for the event origin (e.g., `"sinex-desktop-satellite"`, `"sinex-pkm-automaton"`).
    *   `event_type TEXT`: Type string for the event (e.g., `"window_focused"`, `"note_updated"`).
    *   `ts_ingest TIMESTAMPTZ`: Timestamp of ingestion into this table (database server time). This is the primary partitioning key for TimescaleDB.
    *   `ts_orig TIMESTAMPTZ`: Original timestamp from the source system when the event occurred.
    *   `host TEXT`: Identifier of the machine or device where the event originated.
    *   `ingestor_version TEXT`: Version of the ingestor code that produced this event.
    *   `payload_schema_id ULID`: Foreign key to `sinex_schemas.event_payload_schemas`, identifying the schema for the `payload`. Can be NULL for ad-hoc events.
    *   `payload JSONB`: The core event data. Includes provenance information for derived events showing source event IDs and generating automaton. Correlation is handled through the provenance chain and event relationships.
*   **Referenced TIMs:**
    *   The full DDL and indexing strategy for `raw.events` would be detailed in a TIM like `TIM-EventSubstrateDDL.md` (currently implicitly part of Appendix A of Vision/UG).

### 2.3. TimescaleDB Integration for `raw.events`

To handle the potentially vast volume of time-series data in `raw.events`, TimescaleDB is employed.
*   **Hypertable Architecture:** `raw.events` is converted into a TimescaleDB hypertable, automatically partitioned by the `ts_ingest` column. This significantly improves performance for time-bound queries, data ingestion, and data lifecycle management (e.g., dropping or compressing old chunks).
*   **Benefits:** Efficient time-based queries, optimized ingest rates, native compression for older data chunks (reducing storage footprint).
*   **Referenced ADRs & TIMs:**
    *   `[TIM-TimescaleDBConfiguration.md](docs/tims/data_substrate/TIM-TimescaleDBConfiguration.md)` for detailed setup, hypertable creation DDL, chunk interval guidelines, and compression policy configuration.

### 2.4. Primary Key Strategy: ULIDs via `pgx_ulid`

The primary key for `raw.events` (and other core tables) is the `id` column, using ULIDs.
*   **Architectural Benefits:** ULIDs provide time-ordering crucial for efficient B-tree indexing on `raw.events`, especially given its append-heavy nature. They also ensure global uniqueness suitable for distributed generation scenarios.
*   **Role of `pgx_ulid` Extension:** The `pgx_ulid` PostgreSQL extension provides a native `ulid` data type, efficient binary storage, optimized generator functions (`gen_ulid()`), and casting to/from `timestamp`. This simplifies development and ensures performance.
*   **Referenced ADRs & TIMs:**
    *   `[ADR-001-PrimaryKeyStrategy.md](docs/adr/ADR-001-PrimaryKeyStrategy.md)` for the decision rationale.
    *   `[TIM-PrimaryKeyImplementation.md](docs/tims/data_substrate/TIM-PrimaryKeyImplementation.md)` for `pgx_ulid` setup, DDL examples, and client-side generation.

## 3. Event Schema Management and Validation

> **⚠️ IMPLEMENTATION STATUS: PARTIAL** - Schema registry table ✅ **COMPLETE**, application-level validation ✅ **BASIC**, database-level enforcement ❌ **MISSING**

Managing the structure of the flexible `raw.events.payload` is critical for data quality and interoperability.

### 3.1. Architecture of the Schema Registry (`sinex_schemas.event_payload_schemas`)

A central PostgreSQL table, `sinex_schemas.event_payload_schemas`, acts as the registry for versioned JSON Schema definitions.
*   **Purpose:** Provides documentation, enables optional validation, supports code generation for event types, and manages schema evolution. Each schema is identified by `(event_source, event_type, schema_version)`.
*   **Management Workflow:** JSON Schema files are version-controlled in Git (e.g., `/schemas` directory). A CI/CD pipeline or a dedicated `sinex-schema-manager` agent validates and idempotently loads these schemas into the database table. Changes to schemas are themselves eventified (`sinex.schema.definition_changed`).
*   **Referenced TIMs:**
    *   `[TIM-EventSchemaRegistry.md](docs/tims/data_substrate/TIM-EventSchemaRegistry.md)` for the `event_payload_schemas` DDL, detailed management strategy, and eventification trigger.

### 3.2. In-Database Payload Validation (`pg_jsonschema`)

To enforce data integrity at or near the point of ingestion, the `pg_jsonschema` extension is used.
*   **Architectural Role:** Provides a performant, C-based mechanism for validating `raw.events.payload` JSONB data against its registered schema.
*   **Mechanism:** Typically implemented as a `CHECK` constraint on the `raw.events` table, using `pg_jsonschema`'s `jsonb_matches_schema()` function. This ensures that if a `payload_schema_id` is provided and the schema is active, the payload conforms.
*   **Referenced TIMs:**
    *   `[TIM-EventValidation-pgJsonschema.md](docs/tims/data_substrate/TIM-EventValidation-pgJsonschema.md)` for `pg_jsonschema` setup, the `CHECK` constraint DDL, and performance considerations.

## 4. Satellite Processing Infrastructure

> **✅ IMPLEMENTATION STATUS: OPERATIONAL** - Redis Streams functional, checkpoint system working, automaton framework active, work_queue architecture deprecated

Once events are in `raw.events`, they flow through the satellite constellation for real-time processing via Redis Streams and checkpoint-based state management. The previous work_queue-based architecture has been replaced with Redis Streams for superior performance and durability.

### 4.1. Checkpoint-Based State Management (`core.automaton_checkpoints`)

Automata maintain their processing state through a unified checkpoint system that enables stateful processing with recovery capabilities, replacing the deprecated work_queue approach.
*   **Purpose:** Provides exactly-once semantics, replay capabilities, and fault tolerance for automaton processing in the satellite architecture.
*   **Key Schema Elements:** Stores `automaton_id`, `checkpoint_type` (stream position, file offset, timestamp), `checkpoint_data` (JSONB with type-specific state), `created_at`, `updated_at`.
*   **State Recovery:** Automata resume from their last checkpoint after restarts, using Redis Streams consumer groups and database checkpoints for durability.
*   **Replay Support:** Checkpoints enable historical reprocessing by resetting state and replaying from specific stream positions or timestamps.
*   **Deep Symmetry:** Both ingestors and automata use the same checkpoint system, differing only in checkpoint type (external cursors vs. stream positions).

### 4.2. Redis Streams Message Bus Architecture

Real-time event distribution uses Redis Streams as the central nervous system, completely replacing the previous work_queue polling architecture for superior performance and reliability.
*   **Architectural Pattern:** Events flow from `sinex-ingestd` to a unified Redis Stream (`sinex:events`) where automata consume them using consumer groups, enabling horizontal scaling.
*   **Durability:** Redis Streams provide persistent, ordered event logs with automatic consumer group state management, eliminating the complexity of work_queue claiming and status tracking.
*   **Concurrency:** Multiple automaton instances process events concurrently through Redis consumer groups with automatic load balancing and backpressure handling.
*   **Command/Response:** API commands and responses flow through the message bus using correlation IDs for request/response patterns, enabling asynchronous service communication.
*   **Stream Management:** Automatic trimming, consumer group recovery, and stream monitoring provide operational robustness.

### 4.3. Unified Processor Model: Deep Symmetry Architecture

The satellite constellation implements "Deep Symmetry" - a unified processing model where both ingestors and automata are specialized instances of the same `StatefulStreamProcessor` abstraction.
*   **Processing Flow:** All processors consume from streams (external data sources or Redis Streams), maintain checkpoints, and emit events through `sinex-ingestd`.
*   **State Management:** Unified checkpoint system supports both external cursors (file offsets, timestamps) for ingestors and stream positions for automata.
*   **Scalability:** Processors can be horizontally scaled through Redis consumer groups or multiple ingestor instances with checkpoint coordination.
*   **Deep Symmetry Benefits:** Shared SDK reduces code duplication, provides consistent operational patterns, and enables seamless transitions between scanning and streaming modes.
*   **Satellite SDK:** Encapsulates common patterns for lifecycle management, checkpointing, error handling, and metric collection across all satellite services.

### 4.4. Dead Letter Queue (DLQ) Architecture (`core.dead_letter_queue`)

Terminally failed events from satellite processing are moved to a central PostgreSQL DLQ for analysis and recovery, integrated with the checkpoint-based architecture.
*   **Purpose:** Ensures problematic events can be reviewed, analyzed, and replayed after fixes are implemented without losing data.
*   **Schema:** Stores original event ID, failed payload, error details, retry counts, responsible automaton, stream position, and resolution status.
*   **Integration:** Works with Redis Streams and checkpoint system to ensure failed events can be cleanly reprocessed from specific stream positions after automaton fixes.
*   **Operational Pattern:** Failed events are acknowledged in Redis (removing from pending) but logged in PostgreSQL DLQ for analysis and potential replay.
*   **Management:** Provides CLI tools for replaying DLQ items by re-injecting them into Redis Streams and mechanisms for automated retry policies.
*   **Referenced TIMs:**
    *   `[TIM-DeadLetterQueueImplementation.md](docs/tims/data_substrate/TIM-DeadLetterQueueImplementation.md)` for the `core.dead_letter_queue` DDL, retry policy details, replay tool concepts, and Redis integration patterns.

## 5. Knowledge Representation Architecture

The Exocortex builds structured knowledge from raw and processed data. This is primarily represented through core entities, relations, artifacts, and tags.

### 5.1. Core Entities (`core_entities`)

`core_entities` is the central registry for all canonical "things" or "concepts" the Exocortex recognizes.
*   **Architectural Role:** Forms the nodes of the Exocortex Knowledge Graph. Entities can be persons, organizations, projects, tasks, topics, specific software applications, file paths, or even conceptual documents represented by artifacts.
*   **High-Level Schema:** Each entity has a stable `entity_id ULID`, a defined `entity_type`, a `canonical_label` (primary human-readable name), `aliases`, type-specific `properties` (JSONB), `description`, and potentially an `embedding_vector` for similarity.
*   **Population:** Populated by promotion automata (e.g., from `core_artifacts`), named entity recognition automata, or manual user input.
*   **Referenced TIMs:**
    *   A TIM like `TIM-KnowledgeGraphSchema.md` (or `TIM-CoreEntitiesDDL.md`) would provide the full DDL and indexing for `core_entities`. (Currently this DDL is outlined in Vision App A).

### 5.2. Entity Relations (`core_entity_relations`)

`core_entity_relations` defines typed, directed relationships between entities in `core_entities`.
*   **Architectural Role:** Forms the edges of the Knowledge Graph, capturing how entities are interconnected (e.g., `works_on_project`, `mentions_topic`, `depends_on_task`).
*   **High-Level Schema:** Each relation has a `relation_id ULID`, `source_entity_id`, `target_entity_id`, a `relation_type`, optional `properties` for the relation itself, and validity timestamps.
*   **Population:** Created by automata analyzing text, user input, or derived from system events.
*   **Referenced TIMs:**
    *   `TIM-KnowledgeGraphSchema.md` (or `TIM-CoreEntityRelationsDDL.md`) for DDL and indexing.

### 5.3. Artifacts (`core_artifacts`, `core_artifact_contents`)

Artifacts represent significant, often user-curated, conceptual documents or items.
*   **Architectural Role:**
    *   `core_artifacts`: Represents the conceptual item (e.g., a specific PKM note, a web page archive) with a stable `artifact_id ULID`, `artifact_type` (e.g., `pkm_note`, `webpage_archive`, `email_message`), `canonical_identifier` (e.g., unique note title or URL), `current_title`, and `properties`. It points to the current version of its content.
    *   `core_artifact_contents`: Stores the versioned textual content for artifacts (e.g., Markdown for notes, extracted text for web pages). Each version has a `content_id ULID`, links to its `artifact_id`, the `content_text`, `content_hash_blake3`, `content_format`, and capture timestamps.
*   **PKM Note Content (ADR-004):** For `pkm_note` artifacts, content is managed via Yjs CRDTs. `core_artifact_contents` stores Markdown snapshots derived from Yjs, while `core.pkm_note_yjs_deltas` stores the Yjs update blobs.
*   **Referenced ADRs & TIMs:**
    *   `[ADR-004-PKMNoteContentManagementAndSync.md](docs/adr/ADR-004-PKMNoteContentManagementAndSync.md)` for PKM content strategy.
    *   `[TIM-PKMContentCRDT_Yjs.md](docs/tims/ingestors/pkm_email_nvim/TIM-PKMContentCRDT_Yjs.md)` for Yjs-related DDLs and workflow.
    *   A TIM like `TIM-CoreArtifactsSchema.md` for the DDL of `core_artifacts` and `core_artifact_contents`. (Currently in Vision App A).

### 5.4. Tagging Architecture (`core_tags`, `artifact_tags`)

A universal tagging system allows flexible annotation and organization.
*   **`core_tags`:** Defines canonical tags, each with a `tag_id ULID`, unique `tag_name` (potentially hierarchical, e.g., `project.sinex.docs`), `description`, and optional `parent_tag_id`.
*   **`artifact_tags`:** A join table linking `tag_id` to `target_object_id` (which can be an `artifact_id`, `raw_event_id`, `blob_id`, or `entity_id`) and `target_object_type`. Stores who assigned the tag and when.
*   **Referenced TIMs:**
    *   A TIM like `TIM-TaggingSystemSchema.md` for the DDLs of `core_tags` and `artifact_tags`. (Currently in Vision App A).

### 5.5. Embedding Architecture (`artifact_embeddings`, `event_embeddings`, `embedding_cache`)

Vector embeddings enable semantic search and AI context.
*   **Tables:**
    *   `artifact_embeddings`: Stores embeddings for chunks/summaries of `core_artifact_contents`. Links `content_id` to `embedding_vector`, `model_name`, `embedding_name` (for chunk ID).
    *   `event_embeddings`: Stores embeddings for selected textual fields from `raw.events.payload`. Links `event_id` to `embedding_vector`.
    *   `embedding_cache`: Deduplicates embedding generation by storing `input_text_hash_blake3` -> `embedding_vector` for a given `model_name`.
*   **Indexing & Search:** `pgvector` with HNSW indexes (ADR-005). CPU-first strategy (ADR-007).
*   **Referenced ADRs & TIMs:**
    *   `[ADR-005-VectorIndexTypePgvector.md](docs/adr/ADR-005-VectorIndexTypePgvector.md)`
    *   `[ADR-007-LargeScaleVectorSearchStrategy.md](docs/adr/ADR-007-LargeScaleVectorSearchStrategy.md)`
    *   `[TIM-EmbeddingGenerationModels.md](docs/tims/processing_retrieval/TIM-EmbeddingGenerationModels.md)` for DDLs, backfill automaton logic, and chunking.

## 6. Blob Management Architecture (`git-annex` & `core_blobs`)

Large binary objects are managed by `git-annex`, with metadata in PostgreSQL.
### 6.1. Philosophy & Principles (Vision III.4.1)

*   **Content-Addressing:** `git-annex` uses cryptographic hashes (annex keys) for integrity, deduplication.
*   **Metadata in DB:** `core_blobs` stores metadata (`annex_key`, `content_blake3_hash`, MIME type, size, original filenames, user description).
*   **Content in Annex:** Actual binary data resides in the `git-annex` object store.
*   **Location Independence:** `git-annex` tracks content presence across multiple remotes (local disks, servers, cloud).

### 6.2. Role of `core_blobs` as Metadata Registry

This table is the Exocortex's authoritative index for all annexed content, linking Exocortex `blob_id`s to `git-annex` keys and other descriptive metadata.

### 6.3. Integration Workflow Overview

1.  **Ingestion:** Automaton computes BLAKE3 hash of file. Checks `core_blobs` for dedupe. If new, `git annex add file`, then insert/update `core_blobs`. Commit symlink to Git. Log `sinex.blob.ingested` event.
2.  **Linking:** Other Exocortex entities (artifacts, events) reference blobs via `blob_id`.
3.  **Access:** Query `core_blobs` for annex key/path, `git annex get path` ensures local presence, access via symlink.

### 6.4. Scalability and Integrity Considerations

*   **Hierarchical Annex Repositories:** For very large numbers of files, sharded `git-annex` repositories can improve performance. `core_blobs` would track which annex repo a blob belongs to.
*   **Integrity:** Regular `git annex fsck` (results logged as Exocortex events). `annex.numcopies` policy for data redundancy across remotes.
*   **Referenced TIMs:**
    *   `[TIM-GitAnnexLargeFileMgmt.md](docs/tims/operations/TIM-GitAnnexLargeFileMgmt.md)` for `core_blobs` DDL, detailed workflow, scalability, and integrity practices.

The data substrate architecture leverages several advanced PostgreSQL features:
*   **Triggers:** For automated actions like populating the `work_queue` or eventifying schema changes.
*   **Logical Replication (Conceptual for CDC):** Understanding this for potential future data synchronization to external systems (e.g., dedicated vector DBs).

*   **Apache AGE (Optional Graph Extension):** Provides OpenCypher query capabilities as an alternative for complex graph queries, requiring schema synchronization.
